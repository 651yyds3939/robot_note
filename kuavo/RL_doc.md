# Kuavo RL Environment Architecture (`envs` 目录深度解析与高阶炼丹指南)

本目录包含了基于 Isaac Gym 和 Legged Gym 框架二次开发的强化学习核心环境代码。

**⚠️ 核心认知：这只是“世界”，并不是“大脑”！**
强化学习的核心算法本体，完全不在这个 `envs` 文件夹里！

`envs` 目录下的所有文件，不管逻辑多复杂、奖励函数多华丽，它们在整个强化学习框架中只扮演两个角色：
1. **物理引擎（Physics Engine）**：推演机器人的关节运动、碰撞和重力。
2. **规则法官（Referee）**：根据机器人做出的动作，给它打分（Reward）并告诉它周围的环境信息（Observation）。

它只负责提出问题和评判对错，它本身是不具备“学习能力”的。

最底层的 PPO 算法，藏在别的文件夹。我们暂时“封印”底层 PPO 算法，不要研究它，把全部精力砸在 `envs`（世界）和部署上，是绝对明智的。

整个架构绝不是简单的代码堆砌，而是采用了极其优雅且高度解耦的**双树面向对象继承（OOP）**设计。为了实现“数据（参数）”与“逻辑（物理）”的彻底分离，系统被严格划分为**两大家族**：**参数配置家族（The Configs）** 和 **物理环境家族（The Envs）**。

---

## 🌳 核心架构：双树并行的“四代”继承血脉

理解这个框架的第一步，是认清这里的两套血脉图谱。环境（Env）在启动时，会把配置（Config）作为字典直接“吞”进肚子里，实现逻辑对数据的调用。

以下是极其严谨的代码继承逻辑：

### 第一家族：参数配置家族 (The Configs) —— 系统的“操作控制台”
**职责：纯粹的“规则字典”。里面全是固定的数字、范围和开关（不含任何复杂的推演函数）。**

* 👴👴 **太爷爷 (`base/base_config.py`)**: 
  * 极其底层的基类配置，通常只定义最基础的强化学习通用参数框体，极少直接修改。
* 👴 **爷爷 (`base/legged_robot_config.py`)**: 
  * 继承自 `base_config.py`。
  * **作用**：所有有腿机器人的“基本法”。它定义了虚拟宇宙的物理常量和最底层的训练规则。
  * **核心必看**：`class domain_rand`（域随机化）。这里藏着 Sim-to-Real 的核心机密，比如 `randomize_friction`（随机地面摩擦力）、`push_robots`（随机挨踹测试）、`added_mass_range`（模拟真实传感器负载偏差）。
* 👨 **爸爸辈 (位于 `custom/` 目录下)**:
  * 它们全部平级地继承自 `legged_robot_config.py`，根据不同的机型分支开来：
  * `humanoid_config.py`: 通用双足人形机器人的参数（如增加双臂关节限制，修改摔倒高度）。
  * `kuavo_config.py`: 乐聚夸父早期通用机型的参数。
  * `kuavo_s40_config.py`: 夸父 S40 这一特定硬件的参数。
* 👦 **儿子 (`custom/kuavo_s42_config.py`)**:
  * 继承自 `kuavo_s40_config.py`。这是专门针对 **S42 真机硬件** 的特化调参盘。
  * **核心必看**：
    1. `class control`: 定义了 `stiffness` (刚度) 和 `damping` (阻尼)，完美映射真实电机的 PD 参数。
    2. `class scales`: **炼丹师每天最常改的地方！** 定义了追踪速度、姿势标准度（`joint_pos`）、防打滑等所有奖励函数的**加权权重**。

### 第二家族：物理环境家族 (The Envs) —— 系统的“物理执行引擎”
**职责：纯粹的“动作执行器”。里面全是带 `def` 的函数，负责推演物理定律、计算 PyTorch 张量。写好后极少改动。**

* 👴👴 **太爷爷 (`base/base_task.py`)**: 
  * 这是直接对接 Isaac Gym 显存 API 的最底层接口，处理 4096 个环境的创建、设备分配（Device Allocation）。
* 👴 **爷爷 (`base/legged_robot.py`)**: 
  * 继承自 `base_task.py`。
  * **作用**：底层造物主。
  * **核心位置**：`def step(self, actions):`。时间流逝的主循环（接收张量 -> 计算 Torque -> 物理推演一帧 -> 更新观测状态）。
* 👨 **爸爸辈 (位于 `custom/` 目录下)**:
  * 它们全部平级地继承自 `legged_robot.py`：
  * `humanoid_env.py`: 机器人的“田径教练与裁判”。用 `sin/cos` 函数强制划分步态，并包含所有通用的 `_reward_xxx` 打分公式。
  * `kuavo_env.py`: 针对早期夸父机型的环境逻辑。
  * `kuavo_s40_env.py`: 针对 S40 机型的物理约束。
* 👦 **儿子 (`custom/kuavo_s42_env.py`)**:
  * 继承自 `kuavo_s40_env.py`（同时在内部读取 `kuavo_s40_config` 的数据结构）。
  * **作用**：终极硬件护城河。
  * **核心位置**：重写 `_compute_torques()`。防止仿真里算出离谱力矩，在真机部署时把电机憋爆炸。

### 🐺 孤狼模块：独立工具人 (没有继承，直接被调用)
* **`custom/fk.py`** (`class ForwardKinematics`): 正运动学解算器。利用 PyTorch 写的独立模块，用于在神经网络中快速算出末端执行器（比如手或脚）在空间中的确切坐标。
* **`base/lowpass_filter.py`**: 软件低通滤波器。用于模拟真实硬件中传感器（如 IMU 或电机编码器）数据的物理延迟和噪点，让仿真数据更贴近真实物理世界。
* **`__init__.py`**: Python 包的标识文件，负责注册环境名称，使得你可以通过命令行 `--task=kuavo_s42_sk_ppo` 找到对应的类。

## ⚙️ 运行时逻辑：自下而上的数据闭环

当你在终端运行 `train.py` 时，这“两大家族”按以下层级协同工作，在不到 1 毫秒内完成一次循环：

1. **读取字典 (Config Load)**: 系统首先实例化 `kuavo_s42_config.py`，把地形难度、学习率、奖励权重等参数存入内存。
2. **动作下发 (Network -> Son Env)**: PPO 神经网络算出下一步的关节动作，交给 `kuavo_s42_env.py`。子类环境先检查这套动作会不会导致穿模。
3. **步态对齐 (Father Env)**: `humanoid_env.py` 记录下当前处于哪种步态相位（判断现在该迈左脚还是右脚）。
4. **物理推演 (Grandfather Env)**: 动作传到最底层的 `base/legged_robot.py` 的 `step()` 函数。调用 Isaac Gym 底层 API，让机器人与地面发生碰撞，产生真实受力。
5. **加权打分 (Env -> Config -> Network)**: 物理推演结束，`humanoid_env.py` 里的几十个公式瞬间算完各项得分，并提取 `kuavo_s42_config.py` 里设定的对应权重进行相乘：$R_{total} = \sum (Weight_i \times Reward_i)$。总分返回给 PPO 网络用于计算优势并更新策略。

## 🚀 架构师进阶：优先级学习路线图

> **⚠️ 炼丹师警告**：**绝对不要一开始就试图背诵或逐行研究所有的 `_reward_xxx` 奖励函数！** 很多奖励函数只是为了修补某个特定跑姿而打的“临时补丁”。把精力花在刀刃上，优先攻克以下三大核心壁垒：

### 🥇 优先级 1：死磕 Sim-to-Real（域随机化）
**目标**：理解为什么仿真里健步如飞的模型，到了真机上不会抽搐冒烟。
**行动**：
* 深入阅读爷爷类配置 `base/legged_robot_config.py` 中的 `class domain_rand`。
* 重点研究：`pd_system_delay`（如何模拟通信延迟逼迫 AI 学会预判）、`added_mass_range`（如何应对传感器负载偏差）、`friction_range`（应对复杂地面）。
* **价值**：这是目前具身智能公司最核心的技术壁垒，也是你在真实物理世界部署算法的保命符。

### 🥈 优先级 2：打通部署闭环（ONNX 导出逻辑）
**目标**：明白庞大的训练环境是如何被“脱水”的。
**行动**：
* 寻找代码仓库中的 `export_onnx.py` 或类似脚本。
* 研究它是如何剥离掉环境推演代码、剥离掉 Critic 网络，只保留 Actor 网络，并将其转换为只有几百 KB 大小的纯粹多层感知机（MLP）矩阵文件的。
* **价值**：真实的机器人绝不会背着 Isaac Gym 跑，理解模型的轻量化导出是走向工程落地的必经之路。

### 🥉 优先级 3：底层工程接收与执行（C++ Node）
**目标**：搞懂 `.onnx` 大脑是如何控制钢铁躯体的。
**行动**：
* 跳出当前的 Python 训练仓库，去下位机代码仓库（如 `kuavo-ros-opensource`）中寻找加载 RL 模型的 C++ 节点（通常基于 ONNX Runtime 或 TensorRT）。
* 研究它是如何以 1000Hz 甚至更高的频率，执行 `读取传感器 -> 推理前向传播 -> 下发 EtherCAT 扭矩指令` 这一死循环的。
* **价值**：极致考验实时系统工程能力，是将算法思想化为现实生产力的最后一公里。

## 🎯 调参速查字典：遇到问题改哪里？

* **机器人走得像踩高跷，脚抬太高？**
  👉 去 `kuavo_s42_config.py` 的 `class scales:` 里，适当降低 `feet_air_time` 的奖励权重，或者增加耗能（Torques）惩罚。
* **机器人走着走着两腿交叉（像走猫步）然后绊倒？**
  👉 这是经典的步态问题。先去 `humanoid_env.py` 找防交叉的惩罚函数（如 `_reward_feet_distance`），弄懂它的计算阈值；然后去 config 里把这项惩罚的负数权重调大。
* **我想把在平地训练的机器人拉去爬楼梯？**
  👉 去爷爷类的配置文件 `legged_robot_config.py` 里找 `class terrain:`，把 `terrain_type` 从 `plane` 改为 `stairs`，并调整楼梯的 `step_height` 参数。

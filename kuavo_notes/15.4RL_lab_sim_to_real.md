## 📌 第零章：抄作业专属——极速无痛点火指南 (TL;DR)

> 本章纯粹面向实机操作员。默认机器人下位机（主控电脑）中的 `kuavo-robot-deploy` 工作空间已经完全配置妥当（所有的 v49 物理真值与降维参数均已注入到 `kuavo_v45` 文件夹中并完成了编译）。

### 0.2 物理降落与人员安全防线

在按下遥控器之前，物理世界的防线必须 100% 确认就绪，这是防止硬件报废的最后底线：

1. **精准降落与微松 (触发触地开关)**：

* 摇动龙门架绞盘，让机器人双脚**完全平整地踩在地面上**。
* **核心红线**：贴地后，**只允许再往下摇出 2~3 厘米的绳子**。保证钢丝绳微微打弯但不松垮。这既能完美触发底层的 20N 触地判定，又能保留万一失控时的极限防坠能力。

2. **人员绝对禁区 (防劈叉踢伤)**：

* **禁区**：机器人正前方与正后方（Pitch 俯仰轴）严禁站人。
* **站位**：操作员必须站在机器人**侧后方 45 度角**，双手轻轻搭在机器人的左右肩膀或胸壳上，帮它保持躯干绝对垂直（以此校准内部 IMU 陀螺仪的水平零点）。

### 0.3 遥控器 3 秒无痛通关 SOP (接管大脑)

**⚠️ 遥控器初始状态确认：**
开始前，先把遥控器的 **E 拨杆拨到最左边**，**F 拨杆拨到最右边**。

控制权移交的黄金 3 秒，请严格按照以下时序与分工操作：

* **T-Minus 0.0 秒（死守安全锁）**：
* **助手操作**：双手悬停在键盘 `Ctrl+C` 或者遥控器 **E 档拨杆**（物理急停断电）上。一旦听到刺耳异响或剧烈震荡，0.1秒内瞬间切断。
* **T-Minus 0.5 秒（经典起立）**：
* **主操作员**：确认躯干已扶正，果断将遥控器 **F 档** 拨到左边。
* *预期表现*：机器人不会出现“旱地拔葱”的爆冲巨响。它会发出一声低沉绵密的电机发力声，双腿以极其温柔的低刚度缓缓向上推举，稳稳停在 0.87 米的直立高度。
* **T-Minus 3.0 秒（移交 RL 神经网络）**：
* **主操作员**：确认机器人站立平稳、不再晃动后，**立刻松开搭在它肩膀上的双手**（断开外部干扰输入）。
* 瞬间按下遥控器 **B 键**。
* *预期表现*：底层经典控制闭环被瞬间切断，代码中链接的 `.onnx` 神经网络正式接管机体。机器人开始根据策略进行原地踏步或姿态维持。此时轻轻推或者踹机器人机器人会自己顺着这个力进行跨步。

## 📌 第一章：开发环境制备与跨端物理转移 (纯隔离策略)

> **核心工程纪律：** 在前期的 Sim2Sim（仿真到仿真）测试阶段，我们的工作空间内生成了海量的编译缓存、动态链接库文件以及**基于开发机绝对路径的依赖环境**。如果带着 these “基因污染”直接把代码拷入机器人的下位机（主控 NUC），底层通信节点启动时会立刻引发灾难性的 Core Dump（核心转储）报错。
> **唯一解法**：在开发机的 Docker 隔离舱内彻底洗杀所有 Sim2Sim 编译残留，打出绝对纯净的源码包，再通过 U 盘进行物理跨端转移，并在下位机重新释放。

---

### 1.1 隔离舱预处理：清理 Sim2Sim 历史编译残留

在你的个人开发机上，启动并进入你之前跑仿真测试的 Docker 容器，执行极其严格的文件销毁操作：

```bash
# 1. 进入你本地正在运行的 Kuavo 开发 Docker 容器
docker exec -it kuavo_prep_env /bin/bash

# 2. 进入部署工程根目录 (路径视你的实际挂载情况而定)
cd /workspace/kuavo-robot-deploy

# 3. 毫不留情地删除所有 Sim2Sim 阶段留下的编译产物与隐藏环境标识
rm -rf build/
rm -rf devel/
rm -rf logs/
rm -f .catkin_workspace

# 4. 确认清理完毕后，退出 Docker 容器回到开发机本地终端
exit

```

---

### 1.2 跨端物理转移 SOP：工程打包与 U 盘拷贝

回到开发机的宿主机终端后，将这份“绝对纯净”的源码进行打包，并完成物理隔离传输。

**⚠️ 压缩格式警告：** 必须使用 `tar` 打包为 `.tar.gz` 格式。绝对禁止在 Windows/Ubuntu 下使用 `.zip` 格式压缩！Zip 会导致底层脚本的可执行权限（`chmod +x`）全部丢失，进而导致下位机启动时大量 `launch` 节点因无权限按拒绝执行。

```bash
# 1. 在本地终端进入工作区上一级目录
cd ~/kuavo_dev_workspace

# 2. 将纯净工程打包为 tar.gz
tar -czvf kuavo-robot-deploy-clean.tar.gz kuavo-robot-deploy/

# 3. 将生成的压缩包移动到你的物理 U 盘挂载路径下
cp kuavo-robot-deploy-clean.tar.gz /media/$USER/你的U盘名称/

```

拔出 U 盘。此时，你手里拿着的是一份不受任何历史绝对路径和缓存污染、随时可以在真机上无痛展开的“纯净火种”。

---

### 1.3 下位机环境部署与源码释放 (环境重组)

拿着装有 `kuavo-robot-deploy-clean.tar.gz` 的 U 盘，走到机器人实机旁，将其插入机器人下位机（主控电脑）的 USB 接口。打开下位机的终端，将这把“火种”释放到机器人的主目录中。

```bash
# 1. 将压缩包从 U 盘物理拷贝到下位机的根目录 (通常为 ~ 目录)
# 注意：此处的挂载路径视下位机系统实际识别而定，比如 /media/lab/USB
cp /media/lab/你的U盘名称/kuavo-robot-deploy-clean.tar.gz ~/

# 2. 进入下位机主目录
cd ~/

# 3. 释放工程源码包 (保持原有的权限与目录结构)
tar -xzvf kuavo-robot-deploy-clean.tar.gz

# 4. 彻底删除压缩包，释放下位机宝贵的固态硬盘空间
rm kuavo-robot-deploy-clean.tar.gz

```

至此，下位机中已经躺好了一份绝对干净、零污染的强化学习部署包。底层的物理转移操作圆满结束，下一步即可挂载神经网络模型并重构控制逻辑。

## 📌 第二章：下位机环境部署与神经网络 (ONNX) 挂载

> **深度解耦思想：** 神经网络（`.onnx` 策略文件）作为强化学习的核心“大脑”，必须在下位机中拥有独立、规范的归档路径。我们绝对不能采取在代码中写死路径或频繁更改模型名称的方式进行调试。
> 必须打通 C++ WBC 驱动与 Python 推理环境之间的接口协议，将网络模型、观测维度（87 维）、部署频率（50Hz）在下位机端进行硬核咬合。

### 2.1 神经网络模型 (`.onnx`) 的物理归档规范

在上一章我们将纯净源码释放到下位机主目录后，必须严格遵循下位机的文件寻址规范来存放训练好的策略文件（Policy），防止节点因找不到模型而抛出 `std::runtime_error` 导致程序 Core Dump。

**Step 1: 创建专属模型归档目录**
由于我们在后续操作中需要借用系统自带的 `kuavo_v45` 目录进行“借尸还魂”编译启动，因此必须将模型物理归档到该版本文件夹下的 `rl/models` 子目录中：

```bash
# 确保在下位机工作空间的 config 下为 v45 的 rl 模块创建专属 models 文件夹
mkdir -p ~/kuavo-robot-deploy/src/humanoid-control/humanoid_controllers/config/kuavo_v45/rl/models/

```

**Step 2: 拷贝策略文件**
将你利用 Isaac Gym 训练出的端到端强化学习策略文件（例如 `policy_s42.onnx`）通过 U 盘拷贝至该规范路径：

```bash
# 执行物理拷贝
cp /media/lab/你的U盘名称/policy_s42.onnx ~/kuavo-robot-deploy/src/humanoid-control/humanoid_controllers/config/kuavo_v45/rl/models/

```

### 2.2 配置文件接口打通：修改 `flat_csp.info` / `skw_rl_param.info`

把 `.onnx` 文件放进去只是第一步，底层 C++ 控制节点（WBC 求解器）在按下遥控器 **B 键**（切换到强化学习控制）时，需要通过解析 `.info` 配置文件来动态加载这个网络并匹配输入输出。

请使用下位机中的文本编辑器（如 `vim`），对神经网络的“翻译官”文件进行行级参数对齐：

#### 🔧 1. 精准修改 `flat_csp.info` (或 `skw_rl_param.info`)

**文件路径**：`~/kuavo-robot-deploy/src/humanoid-control/humanoid_controllers/config/kuavo_v45/rl/flat_csp.info`

打开文件，找到并修改以下三大核心神经接口字段，绝对不可漏掉任何一项：

```text
; ==============================================================================
; 强化学习端到端网络加载接口配置
; ==============================================================================

; 1. 神经网络策略文件的绝对寻址路径（必须保证全路径完全正确）
networkModelFile       /home/lab/kuavo-robot-deploy/src/humanoid-control/humanoid_controllers/config/kuavo_v45/rl/models/policy_s42.onnx

; 2. 严格核对并锁定单个周期的观测维度 (Observation Dimension)

; 3. 策略网络输出的动作缩放因子 (Action Scale)

; 4. 关节控制模式对齐
; 2代表位置控制(Position Mode)，必须与我们在 reference.info 中设定的 Mode 2 保持一致
JointControlMode       2

```

### 2.3 性能参数与硬件极限对齐 (i9纯CPU高频推理架构)

在当前“i9 处理器 + 64GB 内存”的顶配无显卡（纯 CPU 推理）工控机架构下，资源总量（RAM）极其充裕，传统的 OOM（内存溢出）危机被彻底消除。但我们面临着更严峻的底层挑战：**算力的绝对稳定性与超低时延的抢占式调度。**

将强化学习策略放进物理真机时，必须在静态下进行以下全方位的性能锁死核对：

#### 📊 1. 双频咬合：50Hz (大脑) vs 1000Hz (小脑)

* **策略推理（50Hz 大脑）**：CPU 端的 ONNX Runtime 负责接收 87 维观测向量，执行一次前向传播，输出下一步的 Action 增量。在 i9 的算力下，单次推理耗时通常仅需 1~3ms，绝对远低于 20ms 的理论上限。
* **物理驱动（1000Hz 小脑）**：底层 C++ 全身力控（WBC）和 EtherCAT/CAN 伺服通信线程，必须以 1ms 的极高频严格准时刷新。
* **致命威胁 (时延抖动 Jitter)**：如果 Linux 默认的完全公平调度器（CFS）在某一毫秒把 1000Hz 的通信线程挂起，转而去执行系统后台任务或 Python 推理，下位机驱动板就会发生**丢帧（Packet Loss）**。真机表现为：机器人明明站得好好的，突然某条腿剧烈抽搐一下。

#### 💻 2. 纯 CPU 架构下的部署优化铁律

针对 i9 纯 CPU 推理架构，在正式点火前，必须严格执行以下配置以压榨出极致的实时性：

| 性能维度 | 极限监测指标 | 纯 CPU 架构最佳实践方案 |
| --- | --- | --- |
| **RAM (系统内存)** | 运行峰值 $\approx$ 4~6GB<br>

<br>空闲空间 $\ge$ 50GB | 64GB 的海量内存在纯 CPU 部署中毫无压力。重点在于确保 `kuavo-robot-deploy` 节点运行期间，**不要开启任何图形化界面工具**（如 RViz, rqt，它们应在上位机或另一台调试电脑上通过 ROS_MASTER_URI 远程开启），让 i9 的内存带宽全部留给底层数据矩阵交互。 |
| **ONNX 并发控制** | CPU 使用率毛刺限制 | **核心锁死**：由于是纯 CPU 跑模型，必须修改 Python 推理脚本或 ONNX Runtime 启动参数，强制指定 `intra_op_num_threads=1` 和 `inter_op_num_threads=1`（或者限制为 2~4 个核心）。如果不限制，ONNX 可能会在 50Hz 的瞬间调用 i9 的所有几十个逻辑线程去算矩阵，这会瞬间阻塞 1000Hz 驱动线程，导致机器人高频打架。 |
| CPU 绑核隔离<br>

<br>(Core Affinity) | 实时响应延迟 $\le$ 1ms<br>

<br>(零丢帧) | 必须通过操作系统的 `taskset` 命令，或者在代码中使用 `pthread_setaffinity_np`，实现**脑干分离**：<br>

<br>1. 将 1000Hz 的通信与 WBC 解算强制绑定到 i9 的**核心 0~3**。<br>

<br>2. 将 50Hz 的 Python ONNX 推理强制绑定到**核心 4~7**。<br>

<br>3. 并将底层的系统优先级提权（`chrt -r 99`），杜绝抢占！ |

通过这套针对 i9 纯 CPU 架构的性能约束，我们能够确保 50Hz 的神经网络预测 and 1000Hz 的底层通信像精密齿轮一样完美咬合，从根本上杜绝“突然抽筋”的物理炸机风险。

## 📌 第三章：版本跨代陷阱与基座包抉择 (环境试错录)

> **研发背景纪实：** 强化学习的 Sim2Real 部署绝不是改个路径就能跑的。当我们把 `.onnx` 模型放入下位机后，立刻迎头撞上了一个极其尖锐的“三维错位”矛盾：**我们在仿真里训练神经网络用的是 v42 模型，官方提供的 RL 部署包骨架是 v46 版本，而站在我们面前的物理真机却是 v49 版本！**
> 在盲目试错之前，我们利用 AI 工具进行了全版本源码级解剖；而在随后的物理测试中，为了绕过系统的启动白名单，我们经历了一系列令人心跳骤停的“暴力试错”，最终才逼近了底层的控制真相。

---

### 3.1 Cursor 全图鉴代码审查：穿透 10+ 历史版本的底层“大乱斗”

在动手修改任何代码之前，为了彻底摸清下位机里堆积如山的配置底细，我们在个人开发机中唤醒了 Cursor（AI 代码审查工具）。

当时工控机里的情况极其混乱：从我们用来训练的 `v42`、到带有各种后缀的 `v45`、官方部署专用的 `v46`、真机出厂的 `v49`，再到所谓的极客尝鲜版 `v50`，甚至还有混杂在各个分支里的 10 多个不知名的魔改历史版本。

* **左摇杆**：
* **前后方向推移**：控制机器人以网络策略进行向前行走或向后退行。
* **左右方向推移**：控制机器人进行左向横移或右向横移（Lateral Motion）。
* **右摇杆**：
* **左右方向推移**：控制机器人绕偏航轴（Yaw）进行原地左转弯或右转弯。

#### 3. 关键物理拨杆功能 (生死死锁线)

* **F键 (最右侧拨杆)**：
* **物理初始状态（右边）**：悬挂准备阶段。
* **拨向左边**：触发底层经典控制的状态机切换，开始执行 `standUpJointAccelTask`，启动平稳直立。
* **E键 (最左侧拨杆)**：
* **物理初始状态（左边）**：安全使能运行状态。
* **拨向右边**：**全局节点紧急退出（急停断电锁）**。一旦遭遇突发状况，将该拨杆推到最右边，系统将瞬间杀掉下位机所有的 ROS 核心控制节点，切断电机使能，机器人瞬间失去全部力矩。

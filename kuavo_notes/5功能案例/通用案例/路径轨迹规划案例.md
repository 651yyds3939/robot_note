## 描述
- 功能包：`trace_path` (位于下位机代码仓库): `/home/lab/kuavo-ros-opensource/src/demo/trace_path`
使用 MPC 控制器跟踪给定的固定轨迹示例，包括:
- 直线：从机器人当前的位置和朝向为起点，生成指定长度的轨迹
- 正方形：从机器人当前的位置和朝向为起点，逆时针生成指定长度的轨迹
- 圆形：从机器人当前的位置和朝向为起点，逆时针生成指定半径的轨迹
- “S” 曲线：从机器人当前的位置和朝向为起点，生成正弦曲线轨迹
没问题，我们把这个基于 ROS 的机器人控制系统想象成一个**“滴滴打车”**的过程。

整个流程可以分为四个核心步骤。我们顺着机器人的“大脑”数据流向，一步步拆解它到底用了哪些文件，以及这些文件都在干什么。

### 第一步：下达任务与规则（相当于“在 APP 上发订单”）
**核心文件：** `follow_circle_traj.launch`[cite: 2]

在 ROS 系统中，`.launch` 文件就像是一个总开关和配置清单[cite: 1, 2]。当你敲下启动命令时，系统首先读取的就是这个文件。

*   **它做了什么：** 它负责拉起相关的程序，并把我们要的“规则”告诉系统。例如，它设定了 `kRadius = 1.0`（告诉机器人要走一个半径 1 米的圆），并设定了 `max_linear_vel = 0.20`（规定最高车速不能超过 0.2 m/s），以及角速度限制 `M_PI/8`[cite: 2]。同时，它还决定了是否要打开 `rviz` 这个可视化监视器来观看整个过程[cite: 2]。
*   **💡 通俗例子：** 就像你在滴滴 APP 上下了一单。你输入了要求：“师傅，去绕着广场转个圈（半径 1 米），别开太快（限速 0.2），顺便把行程分享给我看（打开 RViz）”。

### 第二步：规划死板的路线图（相当于“高德地图生成导航蓝线”）
**核心文件：** `circle_path_generator.cpp`[cite: 1, 3]

接收到任务后，系统需要把“画个圆”这个抽象的概念，变成机器人能听懂的具体坐标点。

*   **它做了什么：** 这是个纯数学计算模块。它首先获取机器人当前的位置和车头朝向，然后根据设定的半径，推算出圆心在哪里[cite: 3]。接着，它利用速度 `speed` 和时间间隔 `dt` 算出每一步该走多远，通过一个 `while` 循环，一步步计算出圆圈上的几百个坐标点，并把每个点的期望坐标 $(x, y)$ 和车头朝向封装成 `PoseStamped` 格式，连成一条完整的路径[cite: 3]。为了严谨，它还在最后做了一次校验，确保起点和终点完美闭合[cite: 3]。
*   **💡 通俗例子：** 导航软件的后台。地图软件根本不管你开的是大众还是奔驰（不管机器人的物理结构），它只是根据几何公式，在虚拟地图上画出了一排密密麻麻的“虚拟面包屑”，告诉你：“只要沿着这些面包屑走，就能画出一个完美的圆”。

### 第三步：老司机开始开车（相当于“司机踩油门和打方向盘”）
**核心文件：** `mpc_path_tracer.cpp` （在 `路径轨迹规划案例.md` 的目录结构中提及）[cite: 1]

现在有了导航路线（面包屑），但机器人自己是不会动的，需要一个控制器来给轮子下发电流指令。这就是 MPC（模型预测控制）的任务。

*   **它做了什么：** 这个文件是真正的“驾驶员”[cite: 1]。它死死盯着上一步生成的圆圈路线，同时感知机器人当前到底在哪里。它会“预测”未来几秒钟的运动，然后精打细算：我现在该给左轮多少速度、右轮多少速度，才能最平滑地切入那条圆弧，并且绝对不能超过第一步里设定的最大速度和最大角速度[cite: 1]。
*   **💡 通俗例子：** 这是一个经验丰富的老司机。他眼睛看着导航上的蓝线（路径点），脑子里想着车辆的性能，然后手脚并用，轻打方向盘、缓踩油门，努力让车子严丝合缝地压在导航蓝线上行驶，遇到急弯还会提前减速。

### 第四步：底盘响应与监控（相当于“车子动起来，并在屏幕上显示”）
**核心文件：** `路径轨迹规划案例.md` 中提到的 `load_kuavo_mujoco_sim.launch` 或 `load_kuavo_real.launch`，以及 `trace_path_demo.rviz`[cite: 1]

最后，指令传达到了底层，我们需要看看效果。

*   **它们做了什么：** 根据 `路径轨迹规划案例.md` 的说明，你需要开启另一个终端来启动物理仿真器（MuJoCo）或者真实的 Kuavo 机器人硬件[cite: 1]。`rviz` 配置文件则负责在屏幕上渲染出一条漂亮的曲线，并实时显示机器人模型是如何沿着这条线运动的[cite: 1]。
*   **💡 通俗例子：** 物理仿真器/真机文件就是车子的“发动机和轮子”，它们接收到老司机（MPC）的油门指令后开始真实转动。而 RViz 就是你手机屏幕上的“行程实时监控”，你可以直观地看到小车图标有没有偏离路线。

---

总结一下这套系统的分工：
**配置 (`.launch`)** 定规矩 ➔ **生成器 (`.cpp`)** 画出理想路线 ➔ **追踪器 (`.cpp`)** 努力沿着路线开 ➔ **底层 (`sim/real`)** 真实执行动作。

你是想进一步了解这个“导航地图”（`circle_path_generator.cpp`）里具体的 C++ 数学循环是怎么把点算出来的，还是对“老司机”（MPC 控制器）怎么限制速度更感兴趣呢？

## 参数设置
### 路径参数
公共参数：
- speed: 速度，单位 m/s
- dt: 时间间隔，单位 s
即每隔`speed * dt`距离在路径上生成一个点
#### 直线
- length: 长度，单位 m
#### 正方形
- side_length： 边长 ，单位 m
#### 圆形
- radius: 半径，单位 m
#### S 曲线
- length: 总长度的一半，单位 m
- amplitude： 振幅

## 编译
修改 `src/demo/trace_path/src/tests/*_main.cc` 对应文件的的路径参数和控制器参数，然后编译运行：
```bash
cd ~/kuavo_ros_opensource
sudo su
catkin build trace_path # 编译
```

## 启动
**终端1**：启动机器人程序

```bash
source devel/setup.bash # 如果使用zsh，则使用source devel/setup.zsh
# 仿真环境运行
roslaunch humanoid_controllers load_kuavo_mujoco_sim.launch # 启动控制器、mpc、wbc、仿真器
# 实物请运行
roslaunch humanoid_controllers load_kuavo_real.launch # 启动实物节点
```

**终端2**：生成轨迹并跟踪

启动参数:
- `rviz` 可视化参考路径和实时路径(可选): `rviz:=true`

```bash
source devel/setup.bash

# 正方形轨迹跟踪
roslaunch trace_path follow_sqaure_traj.launch

# 圆形轨迹跟踪
roslaunch trace_path follow_circle_traj.launch

# S曲线轨迹跟踪
roslaunch trace_path follow_s_curve_traj.launch
```
